- **Packed structs and bit-level layout control**: Support `packed struct` for exact bit layout without padding, and potentially bit fields. Currently FC emits C structs with C's default layout rules, offering no control over padding or alignment beyond what the C compiler provides. Packed structs would enable memory-mapped I/O, binary protocol parsing, and compact serialization formats.

- **uint64 literals > INT64_MAX silently clamped**: The lexer/parser clamps unsigned 64-bit integer literals that exceed `INT64_MAX` (`9223372036854775807`) to `INT64_MAX` instead of preserving their full value. For example, `10481999409208359077u64` is emitted as `UINT64_C(9223372036854775807)` in the generated C. Discovered while writing deterministic tests for `std::random` ŌĆö the `next_uint64` test had to verify hi/lo uint32 parts separately as a workaround. Likely the internal representation uses `int64_t` instead of `uint64_t` for integer literal values.

- **Same-name struct and module at same scope ŌĆö self-referencing doesn't resolve**: When a struct and module share a name as siblings (e.g., `struct lcg_random` and `module lcg_random` side by side within a parent module), functions inside the module cannot use the module-qualified name to call sibling functions. For example, `lcg_random.step(r)` inside `module lcg_random` fails with "undefined name 'lcg_random'". Workaround: use unqualified calls (`step(r)` instead of `lcg_random.step(r)`). This works but diverges from the `data.fc` pattern where `array_list.grow(l)` is called within `module array_list`. The difference is nesting depth ŌĆö `data.fc` modules are at namespace top level, while `random.fc` modules are nested inside a parent module.

- **Restructure `data.fc` to use correct companion struct pattern**: `stdlib/data.fc` uses `namespace std::data::` with each data structure as its own top-level module (e.g., `module array_list` containing `struct array_list`). This should be restructured to match the pattern established in `random.fc`: a single `namespace std::` with `module data` as the parent, and each data structure as a nested submodule with its companion struct alongside (not inside) the same-named module. For example, `struct array_list` and `module array_list` should be siblings within `module data`, not `struct array_list` nested inside `module array_list`. This affects all five data structures: `array_list`, `slice`, `linked_list`, `hash_dict`, `hash_set`. Existing imports like `import array_list from std::data::` would change to `import data from std::` then `import array_list from data` (or use `data.array_list.make()`). Will require updating all `data_*` test files and any demos that import from `std::data::`.

- **SIMD / vector types (evaluate)**: Investigate first-class vector types for SIMD operations (e.g., `float32x4`, `int32x8`). Would require language-level support ŌĆö the types (`__m128`, `__m256`) aren't C structs and can't be represented through extern, and wrapping intrinsics in function calls defeats inlining. The natural fit is emitting GCC/Clang `__attribute__((vector_size(N)))` typedefs, which get arithmetic operators for free in C. **Use cases**: image/audio processing, physics simulations, batch data transforms, cryptography, neural net inference ŌĆö anywhere you're doing the same math on many values at once. **Counterpoint**: GCC/Clang already auto-vectorize many loops, so FC programs may get SIMD "for free" through the C compiler without any language changes. Explicit vector types only matter when auto-vectorization fails or you need guaranteed vectorization. Low priority unless FC targets performance-critical numeric workloads.
