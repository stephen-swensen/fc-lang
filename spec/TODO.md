- **Union companion pattern conflicts with variant construction syntax**: When a union and module share a name as siblings (companion pattern), `union_name.variant(payload)` is interpreted as module member access rather than union variant construction, because the module lookup takes priority in EXPR_FIELD resolution. For example, with `union shape` and `module shape` as siblings, `shape.point(p)` inside `module shape` fails with "module 'shape' has no member 'point'". The struct companion pattern works correctly because struct literal construction (`name { ... }`) uses a separate resolution path. Workaround: use constructor functions in the module (e.g., `let make_point = (p: inner) -> ...`) instead of direct variant syntax. A proper fix would need EXPR_FIELD to fall through to variant construction when module member lookup fails for a companion union.

- **Restructure `data.fc` to use correct companion struct pattern**: `stdlib/data.fc` uses `namespace std::data::` with each data structure as its own top-level module (e.g., `module array_list` containing `struct array_list`). This should be restructured to match the pattern established in `random.fc`: a single `namespace std::` with `module data` as the parent, and each data structure as a nested submodule with its companion struct alongside (not inside) the same-named module. For example, `struct array_list` and `module array_list` should be siblings within `module data`, not `struct array_list` nested inside `module array_list`. This affects all five data structures: `array_list`, `slice`, `linked_list`, `hash_dict`, `hash_set`. Existing imports like `import array_list from std::data::` would change to `import data from std::` then `import array_list from data` (or use `data.array_list.make()`). Will require updating all `data_*` test files and any demos that import from `std::data::`.

- **SIMD / vector types (evaluate)**: Investigate first-class vector types for SIMD operations (e.g., `float32x4`, `int32x8`). Would require language-level support â€” the types (`__m128`, `__m256`) aren't C structs and can't be represented through extern, and wrapping intrinsics in function calls defeats inlining. The natural fit is emitting GCC/Clang `__attribute__((vector_size(N)))` typedefs, which get arithmetic operators for free in C. **Use cases**: image/audio processing, physics simulations, batch data transforms, cryptography, neural net inference â€” anywhere you're doing the same math on many values at once. **Counterpoint**: GCC/Clang already auto-vectorize many loops, so FC programs may get SIMD "for free" through the C compiler without any language changes. Explicit vector types only matter when auto-vectorization fails or you need guaranteed vectorization. Low priority unless FC targets performance-critical numeric workloads.
