# FC Spec ↔ Implementation Audit — 2026-06-03

A full conformance audit of the FC language specification (`spec/fc-spec.html`) against
the compiler implementation (`src/`). Every part of the spec was decomposed into discrete,
falsifiable claims; each claim was checked by reading the relevant source **and** running
behavioral probes through `fcc`. Every factual discrepancy was independently reproduced by
a second adversarial pass before being recorded here.

- **Scope:** 23 audit units, ~600 claims, all 10 spec parts + appendix.
- **Result:** 35 confirmed discrepancies (0 false positives), 8 judgment calls.
- **Line references:** `spec/fc-spec.html` body is embedded markdown; the `md:N` line
  numbers below are into that markdown body, where `html line = md line + 423`. Verbatim
  quotes are included so passages can be located regardless of line drift.

Categories: **IMPL** = compiler violates a correct spec (fix the compiler). **SPEC** = spec
is factually wrong and the compiler is the intended design (fix the spec). **CONTRA** = the
spec contradicts itself (fix the spec; authoritative side noted). **JUDGMENT** = a real gap
that needs a design decision.

## A. Compiler bugs — spec is right, implementation violates it (IMPL, 11)

### A1 — `int32[N]{...}` element count is never checked  🔴 high
Spec md:2038: *"`{ v1, v2, ... }` — exhaustive element list, count must match declared length
exactly — compile error otherwise … There is no partial-list initialization."*
The compiler checks each element's type but never compares the element count to the declared
size (`pass2.c:3874-3908`). Consequences:
- `int32[3] { 1, 2 }` compiles; element 2 is uninitialized (`.len == 3`).
- `int32[3] { 1, 2, 3, 4 }` compiles and emits `alloca(3*…)` then `_arr0[3]=4` — a write
  **past** the stack buffer. Heap form (`alloc(int32[2]{1,2,3,4})!`) overflows the `malloc`.

No diagnostic in either direction. **Silent memory-safety hole.**
**Fix:** in `EXPR_ARRAY_LIT`, require `elem_count == declared_size` when `elem_count > 0`
(`elem_count == 0` is the legal `{ }` zero-init form); add `.error` tests both directions,
stack and heap.

### A2 — Variadic extern function typedef collides  🔴 high
Spec md:3956 (canonical example): `extern printf: (const cstr, ...) -> int32`.
`emit_fn_type_suffix` (`codegen.c:620-635`) does not incorporate `t->func.is_variadic`, but
`type_eq` (`types.c:223`) treats variadic/non-variadic as distinct types. So
`(const cstr, ...) -> int32` and `(const cstr) -> int32` mangle to the **same** typedef name
yet are not deduplicated → two identical `typedef struct {…} fc_fn_uint8_t_ptr__int32_t;`
lines → duplicate-typedef error under `-Werror`. `stdlib/io.fc:24` declares
`remove: (const cstr) -> int32`, so the spec's own `printf` example fails to compile in **any
program that imports stdlib** (the standard build always links it).
**Fix:** append a variadic marker (e.g. `_vararg`) to the mangled suffix when
`t->func.is_variadic`, or suppress the fat-pointer typedef for variadic extern types.

### A3 — Namespace-aliased type resolves to the wrong namespace  🔴 high
Spec md:2781: top-level types in a namespace are mangled with their namespace prefix "so the
same bare name may be declared in multiple namespaces without collision." But an
`import point as wp from widget::` alias resolves to whichever same-named top-level type
appears **first** in compilation order. `import_scope_lookup_kind` (`pass2.c:355`) uses the
namespace-blind `symtab_lookup_kind` for struct/union aliases, while the sibling
`DECL_MODULE` branch (`pass2.c:354`) correctly uses `symtab_lookup_module(..., ref->ns_prefix)`.
The `ImportRef` stores the correct `ns_prefix` (`pass1.c:121,135`) — it is simply discarded.
Reproduces for struct-lit construction, parameter type position, and union-variant
construction; order-dependent. (Existing test `modules/ns_toplevel_same_name_two_ns` passes
only because it constructs via module functions inside each type's own namespace.)
**Fix:** honor the import binding's target namespace in struct/union/type resolution
(`symtab_lookup_kind_ns` with `ref->ns_prefix`). Add a regression test that constructs and
type-annotates an aliased namespaced type whose bare name also exists in another namespace.

### A4 — Generic function used as a value → uncompilable C  🔴 high
Spec md:2460: *"`<type>` syntax is only valid in call position … A generic function cannot be
passed as a value directly."* `let g = identity` (generic) is accepted by `fcc` with no
diagnostic; `pass2.c:~1986` sets `e->type = sym->type` with no guard, and codegen emits the
nonexistent type `fc_fn_unknown__unknown`. Same failure when passed as an argument
(`apply(identity, 5)`). The spec's wrapper-lambda workaround compiles.
**Fix:** reject a generic-function symbol used outside call position in pass2 (EXPR_IDENT and
argument-binding paths) with a clear FC diagnostic.

### A5 — Infinitely-sized recursive struct leaks a C error  🔴 high
Spec md:1754: a direct recursive field without a pointer "would be an infinitely-sized type
and is a compile error, exactly as for non-generic structs." `fcc` accepts
`struct foo = value: foo` and emits invalid C11 (`struct foo { foo value; };`); the only error
comes from the C compiler. No infinite-size check exists in `pass1`/`pass2`. Also hits mutual
value-cycles (`struct a = b_field: b` / `struct b = a_field: a`).
**Fix:** add an infinite-size detection pass: for each struct, walk by-value field types
transitively; error if a struct reaches itself without going through a pointer/slice. Pointer
and slice fields already break the cycle correctly.

### A6 — Unsupported operator in a transitively-instantiated generic leaks to cc  🔴 high
Spec md:2597: errors for unsupported operations on a concrete type are reported "at the **call
site** … Errors are not reported inside the generic definition itself." A **direct** call
`add("x","y")` (with `add = (a:'a,b:'a)->a+b`) is correctly rejected at the FC call site. A
**transitive** call (`wrapper("x","y")` where `wrapper` calls `add`) leaks to the C compiler
(`invalid operands to binary +`). `validate_generic_body`'s `EXPR_CALL` case
(`pass2.c:1445-1449`) recurses only into the call's syntactic children, never into the
resolved callee's body with propagated type bindings. Reproduces for `<` on str, `&` on float.
**Fix:** mirror `mono_discover_transitive`'s fixpoint walk in the validation pass — when a
callee resolves to a generic function, compute its bindings from the substituted argument
types and recurse into its body, propagating the call location.

### A7 — `\xNN` escape does not require exactly two hex digits  🟡 med
Spec md:352: *"`\xNN` … exactly two hex digits."* The lexer accepts 0/1/2 digits via two
independent `if (isxdigit) advance` checks (`lexer.c:314-317` strings, `405-408` chars). Knock-on:
`'\x4'` silently yields byte `0` (`parser.c:500` `length < 6` short-circuit); `"\x4h"`
miscounts `.len` (`codegen.c:1592` hardcodes `j += 3`); `"\x"` leaks a raw cc error.
**Fix:** require exactly two hex digits after `\x` in str/cstr/char literals; emit a diagnostic
otherwise (makes the `parser.c` and `codegen.c` assumptions correct by construction).

### A8 — Unary `-x` accepted on unsigned operands  🟡 med
Spec md:874: negation is valid on "signed integer types, float types." The variable path
(`pass2.c:2225-2231`) checks only `type_is_numeric(ot)` (which includes unsigned), so `-a`
for `a: uint32` compiles and wraps; the literal path correctly rejects `-5u32`. Internal
inconsistency + spec violation.
**Fix:** require `type_is_signed(ot) || type_is_float(ot)` in the prefix-minus path.

### A9 — Inline `cstr` extern constant emits an unparenthesized boundary cast  🟡 med
Spec md:3668: cstr-typed constants get a `char* → uint8_t*` boundary cast. `codegen.c:2078`
emits `(%s uint8_t*)(%s)` with no outer parens, so `(int32) c.my_str[0]` becomes
`(const uint8_t*)((MY_STR)[0])` — the `[0]` binds to the raw literal, then an int→pointer cast
→ cc error. Binding to a `let` first works.
**Fix:** wrap the emitted cast: `((%s uint8_t*)(%s))`.

### A10 — `defer` of a statement-emitting expression → uncompilable C  🟢 low
`defer loop …`, `defer for …`, and block-form `defer if …` are valid FC (loop/for/if are
expressions, defer accepts any expression). `emit_scope_defers` (`codegen.c:110-116`)
unconditionally wraps every deferred expr in `(void)(…)`, but those kinds emit C **statements**
→ `(void)(while(1){…})` cc error.
**Fix:** in the defer path, detect statement-emitting expr kinds and emit them bare, mirroring
the statement-position special-casing already at `codegen.c:1010-1019`.

### A11 — Transitive type-property error attributed to definition, not call site  🟡→🟢 low
Spec md:2597 (same guarantee as A6, property-access variant). For a transitive instantiation,
`'a.nan` errors point at the generic **definition** with no call-site / substitution context,
where the direct case correctly reports the call site. Surfaces via `codegen.c:2111` instead
of the call-site-attributed `pass2.c:1410` path. Lower priority (error is at least surfaced at
the FC level). Same root cause as A6.

## B. Spec is wrong or self-contradictory — implementation is correct (SPEC/CONTRA, 24)

### Self-contradictions (fix the spec; authoritative side noted)

- **B1 (md:1903, 🔴):** "no null comparison … null is not expressible" contradicts the Default
  Values table (md:3312, `T*` default = "physically null"), pointer equality (md:836), and the
  spec's own `default(T*)` examples (md:1922/3422/3936/3946). Qualify md:1903: there is no null
  *literal*, but a null raw pointer is obtainable via `default(T*)` and comparable by address.
- **B2 (md:223, 🟡):** type-var first char — prose says letter/digit/underscore, `grammar.bnf:103`
  says letter/underscore, impl allows letter only. Align all to letter-first.
- **B3 (md:389 vs 391, 🟡):** `%.2f{pi}` listed as allowed, but `%f` "requires explicit width";
  impl enforces the requirement. Change the example to `%8.2f{pi}`.
- **B4 (md:1633-1637, 🟡):** closure-capture example places `let threshold`/`let mut counter`
  at top level with `// captures by copy` / `// compile error` comments, contradicting md:1628
  ("top-level bindings are not captures") and the md:1640 `total` example. Make the example
  bindings local.
- **B5 (md:2926, 2938, 🟡):** the §Non-forwarding and §Shadowing snippets place file-level
  imports **after** `module` declarations, violating the imports-first rule (md:2855) the
  compiler enforces. Reorder the snippets.
- **B6 (md:3238, 🟡):** the `<stdio.h>` trigger is written `"\{expr}"`, contradicting the real
  `%spec{expr}` interpolation syntax (md:360/409); `\{` is an "unrecognized escape." Fix the cell.
- **B7 (md:3533, 🟢):** comment "same as `alloc(int32[4])`" contradicts md:3473 ("the bare form
  `alloc(T[N])` is not valid"). Drop the reference.

### Factual errors (fix the spec; impl is the intended design)

- **B8 (md:227, 🔴):** reserved-words list omits `const`, `when`, `defer` — all real keywords
  (`lexer.c:103/113/124`), also listed in `grammar.bnf:111-116`. Add them.
- **B9 (md:3224-3244, 🔴):** "5-header preamble" / 5-row core-header table — codegen emits **7**
  unconditionally; `<assert.h>` and `<limits.h>` are load-bearing (`assert(...)`, `SIZE_MAX`/
  `INT_MAX` in always-emitted helpers). Add two rows; change "5" → "7".
- **B10 (md:46, 🟡):** `_Static_assert` "compiler-generated" — never emitted (`grep` finds none
  in `src/`); only programmer-authored extern headers use it (as md:3735/3766/3806 describe).
- **B11 (md:50-53, 🟡):** "generated C uses two extensions beyond C11" undercounts — every
  program emits `__attribute__((unused))`/`((constructor))`, and bounds checks emit
  `__builtin_expect`. Enumerate the full set (statement exprs, `__builtin_alloca`,
  `__attribute__` family, `__builtin_expect`).
- **B12 (md:237-239, 🟡):** built-in type names "may not be used as binding/type names" — not
  enforced; `let int32 = 5`, `struct any` compile. Soften to descriptive, or enforce.
- **B13 (md:1714, 🟡):** storing a capturing closure into heap "undefined behavior — not
  currently detected" — the compiler **does** detect and reject it (`pass2.c:160, 4250, 3155`),
  mirroring the adjacent "returning a capturing closure" rule. Change to "compile error."
- **B14 (md:2030 and md:3369, 🟡):** `uint8[3] { 255, 0, 128 }` does not compile — int literals
  don't narrow to `uint8` elements (per the spec's own md:34 "never reinterprets a literal").
  Use suffixed literals (`255u8, 0u8, 128u8`).
- **B15 (md:2325-2329, 🟡):** Static Type Properties table omits `isize`/`usize`, which fully
  support `.min`/`.max`/`.bits`. Add a row.
- **B16 (md:2711, 🟡):** "errors only from passes 1/2" — code generation (stage 5) also
  accumulates errors and exits 1 (transitive type-props at `codegen.c:2111`; conflicting
  `#define` at `codegen.c:4374`), and deletes the partial output (undocumented). Amend.
- **B17 (md:3161, 🟡):** "inactive branches receive no checking of any kind" — lexical checks
  (tabs, unexpected chars, unterminated strings/comments) still fire (tokenization precedes
  branch stripping, `lexer.c:853` before `:857`). Only parsing/semantic checks are skipped.
- **B18 (md:4150, 🟡):** `std::data.array_list.make()` "direct access" — `::` in expression
  position is a parse error; `::` is import/namespace-only. Remove the claim or show
  `import data from std::` then `data.array_list.make()`.
- **B19 (md:3893-3896, 🟡):** `extern open: (cstr, int32)` + `posix.open(c"file.txt", 0)` is
  rejected — `c"…"` is `const cstr` and the compiler won't strip const. Declare the param
  `const cstr` (consistent with md:3914).
- **B20 (md:180, 🟢):** tabs rejected only "at the start of a line" — impl rejects tabs anywhere
  outside strings/comments (`lexer.c:565-568`), which is safer. Broaden the wording.
- **B21 (md:879, 🟢):** cast table "Valid on: numeric types" contradicts the spec's own §Casting
  section (md:432-480), which already documents bool→num, ptr↔ptr, ptr↔int, str↔cstr, slice
  const-cast. Make the summary row cross-reference §Casting.
- **B22 (md:3497, 3539, 🟢):** transpilation notes say `calloc` for literal-initialized `alloc`
  forms, but codegen emits `malloc` + full write (`alloc(point{…})`, `alloc("…%d…")`,
  `alloc(int32[4]{1,2,3,4})`). Bare-type/raw-buffer/empty-array forms **do** use `calloc`.
- **B23 (md:4119, 🟢):** `str { ptr = cs, len = libc.strlen(cs) }` doesn't type-check when
  `strlen → usize` (the spec's own recommendation) — `usize` doesn't widen to `str.len`
  (int64). Show a `(int64)` cast.

## C. Judgment calls — design decision needed (8)

- **C1 (md:221):** the `__` double-underscore ban applies to identifiers but not type vars
  (`'a__b` is accepted). Extend the ban, or document the exemption?
- **C2 (md:464-471):** `(str) cstr` is supported and sound (strlen-wrap, shares backing) but
  §Casting only documents `str → cstr`. Document the inverse?
- **C3 (md:436-444):** `(bool) bool` identity cast is rejected ("invalid cast"); numeric
  identity casts are allowed. Document bool∉numeric, or allow identity casts uniformly?
- **C4 (md:497-498):** self-assignment `x = x` is a compile error but undocumented; only bare
  idents are caught (`buf[0] = buf[0]` and `p.x = p.x` are allowed). Document, or widen?
- **C5 (md:1758):** duplicate field in a struct literal (`point{ x=1, x=2 }`) is silently
  last-wins and emits override-init C (fails `-Wextra`). Reject, or spec last-wins?
- **C6 (md:1987-1992):** §Pointer-arithmetic states no rules; impl allows `ptr±int` only
  (`ptr-ptr`, `int+ptr`, `ptr+ptr` rejected) and is unchecked. Document the legal forms?
- **C7 (md:2558):** `option<T>` is accepted as an unresolvable user type and leaks a C error
  rather than a clean FC diagnostic (general unresolved-type-name gap). Reserve `option`?
- **C8 (md:3856):** a reserved identifier (`free`, `default`, …) is accepted as an extern decl
  name (unreferenceable) despite the "cannot be used anywhere" rule — and the §any* example
  (`extern free: (any*) -> void`) relies on the lenient behavior. Reconcile rule vs. example.
