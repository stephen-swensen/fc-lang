## P0 ‚Äî Silently wrong runtime behavior from plausible code

### 1. Narrow-width arithmetic leaks C integer promotion (codegen.c ~1767‚Äì1855) ‚Äî ‚úÖ FIXED 2026-06-12 (`+ - * <<`; division ‚Üí item 15)
- `200u8 + 100u8 == 44u8` is **false** (operands promote to `int`, sum = 300), while
  `let s = a + b; s == 44u8` is **true** (assignment truncates). Unsigned narrow `+ - * <<`
  are emitted plain; the cast-back-through-unsigned pattern is gated on `type_is_signed`.
- `-1i16 * -1i16` is UB: `(uint16_t)` promotes to `int`, `65535 * 65535` overflows
  (UBSan-confirmed). The cast-through-unsigned pattern is only sound at widths ‚â• `int`.
- `-128i8 / -1i8` yields `+128` when consumed directly (promotion avoids the SIGFPE but
  produces an unrepresentable value).
- **Fix:** cast the result back to the FC result type for *all* sub-`int` arithmetic
  (signed and unsigned); route 8/16-bit multiply through `unsigned`:
  `(int16_t)(uint16_t)((unsigned)(uint16_t)a * (unsigned)(uint16_t)b)`.
- **Resolution (2026-06-12):** scoped to `+ - * <<` (the promotion-leak operators).
  `unsigned_counterpart` now covers unsigned kinds and a new `type_is_subint` helper gates
  the narrow path. Sub-`int` `+ - *` route through `unsigned` and truncate back (signed and
  unsigned); the multiply no longer hits `int`-overflow UB (re-verified under UBSan). The
  left-shift cast-back was extended from signed-only to signed-or-sub-`int`. Wider signed
  types keep the existing cast-through-counterpart; wider unsigned types stay plain. This
  routing is **`int`-width-agnostic** ‚Äî correct on 16-bit-`int` targets too, since the math
  is modular in `unsigned` and only the low bits are kept. Spec "Integer overflow" section +
  Transpilation note updated. Tests: `expressions/narrow_arith_wrap`,
  `expressions/narrow_arith_consistency` (both confirmed to abort on the pre-fix binary).
- **Deferred to item 15 (not fixed here):** the `-128i8 / -1i8` ‚Üí `+128` symptom. A result
  cast-back makes it *look* fixed on a 32-bit-`int` host, but it is the same `int.min / -1`
  overflow as item 15 surfacing via promotion, and on a target where the FC type is as wide
  as C's `int` (e.g. `int16` on a 16-bit `int`) `int16.min / -1` is a genuine SIGFPE/UB in
  the divide itself ‚Äî a cast-back cannot help. Defining it requires guarding the divide and
  choosing wrap-vs-abort, so it belongs to item 15's decision and is folded there (covering
  `int8`/`int16`/`int32`/`int64`/`isize` uniformly).

### 2. `for` loops: endpoint typing and re-evaluation (pass2.c:4517‚Äì4525, codegen.c:2701‚Äì2730) ‚Äî ‚úÖ FIXED 2026-06-12
- Loop var type picked by **TypeKind ordinal max**, not widening rules: `for i in 0u32..m`
  with `m: int32 = -1` compiles and runs ~4G iterations (`-1` ‚Üí `4294967295`). Mixes that
  `+`/`<` reject are accepted; pass2's type and the emitted C type can differ
  (pass2 says `uint8`, C emits `int64_t`). `0..n` with `n: usize` violates the
  no-implicit-isize/usize rule and has signed-overflow UB on `i++` for huge `n`.
- Range end and slice iterable are re-evaluated **every iteration**: `for i in 0..get_end()`
  calls `get_end()` n+1 times; `for x in get_slice()` calls it 2n+1 times. Spec's reference
  lowering shows a single bound.
- **Fix:** unify endpoints via `type_common_numeric` + `wrap_widen` (reject when no common
  type), emit the unified type in C, and hoist end/iterable into once-evaluated temps.
- **Resolution (2026-06-12):** pass2 now unifies range endpoints with `type_common_numeric`
  and `wrap_widen`s both to the common type (no common type ‚Üí `range endpoints have
  incompatible types` error). Because the start is widened too, `iter->type` *is* the unified
  type, so the pass2/codegen type mismatch is gone ‚Äî `uint32..int32` and any implicit
  `isize`/`usize` mix (e.g. `0..n` with `n: usize`) are now rejected just as `+`/`<` reject
  them. Codegen hoists the range end into `_fe%d` and the slice iterable into `_fs%d` before
  the loop, so a side-effecting bound/iterable is evaluated exactly once (was n+1 / 2n+1).
  Spec "Range iteration" + index-element prose + the Transpilation lowering updated to state
  the common-type requirement and single-evaluation. Tests: `control_flow/for_range_end_once`,
  `for_slice_iterable_once`, `for_range_widen_consistent`, `for_range_signedness_err`,
  `for_range_usize_mix_err`.

### 3. Escape analysis: assignment-sink omissions (pass2.c:3433, 3508‚Äì3568) ‚Äî ‚úÖ FIXED 2026-06-13
The checked sinks are only `EXPR_IDENT` globals and direct `h->field`. Confirmed
stack-use-after-return (ASAN) through four bypasses:
- `let mut p = param; p = &x; return p` ‚Äî provenance recorded at bind time, never updated
  on reassignment (defeats return, heap-field, and global sinks alike).
- `buf[0] = &x` into a heap slice element ‚Äî `EXPR_INDEX` target unchecked.
- `(*h).p = &x` ‚Äî parses as `EXPR_FIELD` over a deref; the equivalent `h->p = &x` is rejected.
- `*hp = &x` through a heap `int32**`, and nested paths `ho->i.p = &x`.
- **Fix:** merge RHS provenance into the binding on assignment (conservatively
  stack-taint), and walk every assignment target's lvalue path to its root, applying the
  heap-store check whenever the root is heap-provenance ‚Äî not just one target shape.
- **Resolution (2026-06-13):** both halves of the fix landed, plus the loop-carried corner
  the four bypasses implied. (a) `assign_dest_prov` walks the assignment target's lvalue path
  (`*p` / `p->f` / `obj.f` / `obj[i]`) to its **root** storage provenance; a heap/static root
  is rejected uniformly, so `h->buf[i] = &x`, `(*h).p = &x`, `*hp = &x`, and `ho->i.p = &x`
  are all caught by one check instead of the single `h->field` shape. A local-stack root stays
  exempt. (b) `scope_taint_prov` merges the RHS provenance into a reassigned mut binding
  (monotone toward STACK), so `p = &x; return p` (and the global/heap-field variants) fire;
  the merge is branch-safe (a stack assignment in one arm taints the join) and conservatively
  rejects reassign-back-to-heap. (c) Because the single textual pass would still miss a read
  that precedes the tainting assignment but follows it across a loop iteration, a fixpoint
  `pretaint_loop_body` pre-taints any in-scope mut binding assigned a (transitively) stack
  value anywhere in a loop body before the body is checked ‚Äî `expr_may_yield_stack` is a
  conservative syntactic predicate (call results excluded: a function can't return stack;
  `&fn` excluded: static fn pointer). The heap-accumulator-in-loop pattern is preserved
  (heap/unknown idents and call results never taint). Intraprocedural latitude is unchanged:
  writes through unknown-provenance pointers (out-params) stay accepted. Spec "Escape analysis"
  generalized (heap/static-root assignment sink, mut-reassignment merge, loop-carried note) +
  the ¬ßPointers summary line. Tests (`tests/cases/escape/`): the four bypasses
  (`assign_mut_reassign_{return,global,heap_field}_err`, `heap_slice_elem_assign_err`,
  `deref_field_assign_err`, `deref_heap_assign_err`, `nested_heap_path_assign_err`), the
  branch + loop-carried corners (`assign_branch_taint_err`, `loop_carried_{return,saved}_err`),
  and four positives (`assign_mut_reassign_heap_ok`, `stack_ptr_into_stack_array_ok`,
  `loop_heap_accum_ok`, `loop_stack_local_use_ok`). All confirmed to compile on the pre-fix
  binary.
- **Follow-up (2026-06-14) ‚Äî widening laundered provenance (the deeper root of the
  item-5 "adjacent gap" note below):** the escape sinks all read `expr->prov`, but the
  implicit-widening wrapper `wrap_widen` (pass2.c) allocated a fresh `EXPR_CAST` node and
  never copied the operand's provenance ‚Äî so the arena-zeroed default `PROV_UNKNOWN` silently
  replaced a `PROV_STACK` taint whenever a stack value was implicitly widened. Adding `const`
  is a lossless widen (`cstr` ‚Üí `const cstr`, `int32[]` ‚Üí `const int32[]`), so a stack
  `(cstr[N]) s` / `alloca(...)` / subslice assigned into a *const* destination ‚Äî a `const cstr`
  global, a heap struct's `const cstr` field, a `const`-typed mut binding later returned, a
  `some(...)` payload widened to a const option, or a heap struct-literal field ‚Äî was widened
  and escaped **every** sink (return, global, heap-field, struct-literal-field, option-payload),
  not just one shape. The item-3 assignment-sink generalization already caught the *non-widened*
  slice/cstr-into-heap-field store (the note's literal `h->name = alloca(c"‚Ä¶")` with a
  non-const field is rejected); the surviving hole was specifically the widen path. **Fix:**
  `wrap_widen` now propagates `cast->prov = e->prov`. A widen either yields a fresh scalar
  (numeric ‚Äî no provenance) or a pointer/slice aliasing the same storage (const-add,
  slice const-change), so copying the operand's provenance is always sound and fixes all sinks
  uniformly at the root rather than per-sink. Audited the other synthetic-`Expr` creators:
  `const_clone_expr` already copies via `*n = *src`; `wrap_widen` was the lone prov-dropping
  wrapper. Spec "Escape analysis" ¬ßProvenance-propagation list now includes implicit widening.
  Tests (`tests/cases/escape/`): seven negatives across the sinks
  (`widen_global_cstr_err`, `widen_global_slice_err`, `widen_return_cstr_err`,
  `widen_heap_field_cstr_err`, `widen_struct_lit_field_err`, `widen_option_payload_err`,
  `widen_alloca_const_field_err` ‚Äî all confirmed to compile on the pre-fix binary) and three
  positives proving no false-positive over-rejection on the widen path
  (`widen_heap_into_const_global_ok`, `widen_heap_into_const_field_ok`,
  `widen_stack_local_use_ok` ‚Äî ASan/UBSan-clean). Full suite green (gcc + clang).

### 4. String interpolation defects (codegen.c:1331‚Äì1432, 1568‚Äì1585) ‚Äî ‚úÖ FIXED 2026-06-13
- `%s{cstr-expr}` evaluates the expression **twice** (strlen + snprintf arg).
- `%x{-16i8}` prints `"ff"` instead of `"f0"`: `(unsigned int)` sign-extends, then the
  2-char width budget truncates the *leading* digits.
- Width budget treats printf minimum-field-width as a maximum: `"%8.2f{1e9}"` ‚Üí
  `"10000000"`; `%10s{c"hi"}` truncates; no-width `%f` bound of 24 truncates large values.
- Segments evaluate out of textual order (str-typed segments pre-evaluated first, the rest
  in C-unspecified snprintf-arg order).
- **Fix:** pre-evaluate *all* non-literal segments into temps in textual order; mask narrow
  `%x` args to operand width before widening; compute `%f`/cstr budgets from actual bounds.
- **Resolution (2026-06-13):** `emit_interp_string_impl` now pre-evaluates **every**
  non-literal, non-`%T` segment into a typed temp (`_sg<tid>_<k>`) in textual order before
  the buffer-size and `snprintf` passes; all three passes index the same temps. This makes
  each interpolated expression evaluate **exactly once** (the cstr `%s` double-eval is gone:
  strlen now reads the temp) and **left-to-right** (no more reliance on C's unspecified
  argument-evaluation order ‚Äî `%T` is excluded since its operand is never evaluated).
  Unsigned conversions (`%x %X %o %u`) now reinterpret the operand at its **own width** via
  the exact-width unsigned counterpart (`(unsigned int)(uint8_t)x`, ‚Ä¶) before widening, so a
  negative narrow operand prints its bit pattern (`%x{-16i8}` ‚Üí `"f0"`) instead of a
  sign-extended value that overran the budget; `int32`/`int64`/`isize`/`usize` are unchanged
  (their printf width already matched their bit width). Buffer budgets are now true upper
  bounds: `%f` is sized as sign + max integer digits (309 for `float64`, 39 for `float32`) +
  `.` + precision (the field width can only raise this floor, never cap it); `%e %g` are
  sized from precision (`9 + prec`) rather than a fixed 15/24 that an explicit precision could
  blow; cstr `%s` honors an explicit field width (was strlen-only ‚Üí padded output overran).
  The buffer-is-a-compile-time-constant property (for loop-hoisting) is preserved: the new
  bounds are still constants when no bare-`%s` is present. Fixed sizes are explicit-width
  casts and absolute digit counts, so the routing is **`int`-width-agnostic**. Spec "String
  Interpolation" updated: intro states once/left-to-right; the `%f` width note corrected
  (minimum field width, never a cap); the Allocation paragraph states field width is a floor.
  Tests (`tests/cases/strings/`): `interp_unsigned_negative` (`%x %X %o %u` of negative
  `int8`/`int16`/`int32`), `interp_float_no_truncate` (`%f` wide magnitude + `%.20e`),
  `interp_cstr_width_pad` (cstr `%Ns` left/right pad), `interp_eval_once` (cstr/str/int single
  eval), `interp_eval_order` (textual order via side effects), `interp_mixed_segments`
  (str/cstr/numeric interleave ‚Äî temp-index alignment). All six confirmed to abort on the
  pre-fix binary; ASan/UBSan- and `-O2`-clean.
- **Follow-ups (design decisions, deferred):**
  - The `%f` explicit-width *requirement* is now vestigial ‚Äî its original justification
    ("fixed-point output can be arbitrarily long," implying width bounds it) was false, and
    the buffer is sized from the type's max regardless. Dropping the requirement (allow bare
    `%f`) is a loosening and thus a design decision; left as-is for now.
  - A correct `%f{float64}` budget is now ~317 bytes (was width-or-24). This is sound but
    large; in a loop it would have compounded the alloca-growth problem from **item 5** ‚Äî but a
    `%f` buffer is constant-size, so item 5's fix (2026-06-13) hoists it to a fixed
    function-entry array reused per iteration: the ~317 bytes are paid once, not per pass. A
    value-aware runtime-tightened bound was explicitly declined (static cost over runtime
    machinery).
  - Interpolation is now deterministically left-to-right / once. This **locally settles** the
    evaluation-order question for interpolation; the broader call-argument and operator
    ordering (**item 19**) was settled the same way ‚Äî left-to-right, evaluate-once ‚Äî on
    2026-06-15 (see item 19).
  - **New finding (not in the original item 4) ‚Äî ‚úÖ FIXED 2026-06-13:** integer interpolation
    was not int-width-agnostic for the 32-bit FC types on a **16-bit-`int`** target.
    `%d{x:int32}` emitted `(int)x` + `%d`, and the format-string `ll` modifier was gated on
    64-bit FC types only ‚Äî so on a target where `int` is 16 bits (and `int32_t` is `long`),
    the `(int)` cast truncated to 16 bits and `%d` read 16 bits: silent wrong output for
    `|value| ‚â• 2^16`. `int8`/`int16` were safe (they fit a 16-bit `int`) and
    `int64`/`isize`/`usize` already used `ll`; only `int32`/`uint32` fell in the gap (same
    int-width-agnosticism class as items 1 and 15). **Resolution:** every *integer-number*
    conversion (`d i u x X o`) is now passed as `(long long)` / `(unsigned long long)` (the
    unsigned path still masks to the operand-width unsigned counterpart first) and the
    format-string always emits the `ll` length modifier for them ‚Äî uniform across all integer
    widths, so the printf type matches on both 16- and 32-bit-`int` targets. `%c` (char/uint8)
    stays an `int` argument with no modifier. Zero observable change on 32-bit-`int` hosts
    (verified: full suite + `tests/cases/strings/interp_int_width`, which pins int32/int64/u64
    decimal/hex/octal/unsigned and `%c`; ASan/UBSan/`-Wformat`-clean). The buffer-size bounds
    were already by-FC-type and needed no change.

### 5. `alloca` for array literals ‚Üí unbounded stack growth in loops (codegen.c:2427‚Äì2436) ‚Äî ‚úÖ FIXED 2026-06-13 (constant-size cases hoisted; runtime-sized cases resolved by the alloca redesign, same day)
`for i in 0..1000000 ‚Üí let buf = int32[64] { }` segfaulted (~256 MB of alloca never freed
until function exit; reproduced at gcc `-O0`/`-O2` and clang `-O0` ‚Äî clang `-O2` happened to
fold the constant-size alloca into one slot, masking it). Interpolated strings and `(cstr)`
casts in loops accumulate the same way.
- **Resolution (2026-06-13):** the backing storage for every *constant-size* stack temporary
  is now a plain fixed C array hoisted to the top of the enclosing function (`g_fn_backings`
  in codegen.c), referenced at the use site instead of `__builtin_alloca`. The single slot is
  reused on every loop iteration ‚Üí bounded stack, no frame growth. Covers (a) **all array
  literals** ‚Äî `N` is always a compile-time literal (enforced in pass2), so the backing is
  `T name[N];`: standard C89, no VLA, no builtin; and (b) **constant-size interpolation
  buffers** ‚Äî when every segment is compile-time-bounded (numeric/char/pointer/`%T`, and any
  `%s` carries an explicit precision) the buffer is `uint8_t name[N+1];`, sized identically to
  the emitted `_flen`. The backing keeps function-frame lifetime, matching the lifetime
  `alloca` already gave the produced slice/str, so escape analysis (item 3) stays valid ‚Äî a
  still-live pointer can't be invalidated. Shared helpers (`interp_numeric_bound`,
  `interp_literal_len`) feed both the emitted `_flen` and the constant-size scan, so the two
  can never disagree on the budget (no under-size ‚Üí no overflow).
- **Deliberately *not* done**, per the project's "static C-like costs over hidden runtime
  machinery" principle: no `__builtin_stack_save`/`__builtin_stack_restore` bracketing (the
  original sketch's "save/restore per iteration" ‚Äî non-standard builtins, dubious on the
  16-bit-`int` embedded targets FC supports) and no value-aware runtime-tightened bounds.
- **Runtime-sized temporaries ‚Äî RESOLVED 2026-06-13 by the alloca redesign (breaking).** Rather
  than leave a hidden `alloca` behind unprecisioned `%s` interpolation and bare `(cstr) str`
  casts, those two forms are now **compile errors** ‚Äî FC has no *implicit* unbounded stack
  allocation. The error forces an explicit home, in increasing cost: a compile-time **bound**
  (`%.Ns`, or the new truncating `(cstr[N])` cast ‚Üí hoisted fixed backing, loop-safe), the
  **heap** (`alloc(‚Ä¶)!`), or **dynamic stack** via the new **`alloca`** operator (parallel to
  `alloc`: no option, no `free`, escape-checked; same `__builtin_alloca` codegen as the old
  implicit path, now reached only through the keyword). Rationale: the project's "static
  C-like costs over hidden runtime machinery" principle taken to its conclusion ‚Äî dynamic
  stack stays available (defensible for small, ergonomics-first programs) but is never
  silent. The `__builtin_stack_save/restore` block-scoped sketch remains rejected (non-portable
  on 16-bit-`int` targets). Stdlib C-interop (`io`/`sys`/`net`/`text`) migrated to
  `alloca(c"%s{path}")`, which is non-truncating and loop-safe from the caller (reclaimed when
  the stdlib function returns). See `spec/fc-spec.html` ¬ßString Interpolation ‚Üí Allocation,
  ¬ßCasts (str‚Üícstr), and ¬ßStack Allocation ‚Üí Dynamic stack (`alloca`).
- **Ergonomic follow-up (2026-06-14):** the single-segment-interpolation spelling of a plain
  str‚Üícstr copy (`alloca(c"%s{path}")` / `alloc(c"%s{path}")!`) read as "format this", not
  "convert", and routed a memcpy through `snprintf`. `alloc`/`alloca` now also **license a bare
  `(cstr)` cast** as their direct argument ‚Äî `alloca((cstr) path)` (dynamic stack),
  `alloc((cstr) path)!` (heap) ‚Äî exactly as they already license an unbounded interpolation
  (the parser's `wrapped`/`licensed` flags). This unifies the model: `(cstr)` is *the
  conversion*, and `[N]` / `alloc` / `alloca` are *the three homes*, applied uniformly to both
  casts and interpolations. It's a direct memcpy (no `snprintf`), adds zero keywords, and keeps
  `c"..."` for genuinely-formatted cstrs. Stdlib migrated to the cast form; `c"%s{...}"` casts
  in interp stay valid for formatting. Tests: `strings/cstr_cast_homes`,
  `strings/cstr_cast_heap_returns`, `escape/return_alloca_cstr_cast_err`.
- **Tests:** array-literal/constant-interp backings as before
  (`slices/array_lit_loop_reuse`, `array_lit_distinct_backings`, `strings/interp_loop_reuse`,
  `generics/generic_array_lit_backing`); plus the redesign: `strings/cstr_bounded_cast`
  (fit/truncate/exact/empty), `strings/cstr_bounded_loop` (200 k loop, one backing slot),
  `memory/alloca_interp`, `memory/alloca_buffer` (raw + slice), `generics/generic_cstr_bounded`
  (per-monomorph backing), and the negative tests `strings/interp_unbounded_err`,
  `strings/cstr_cast_unbounded_err`, `strings/cstr_cast_literal_err`,
  `casts_widening/cstr_buffer_nonstr_err`, `escape/return_alloca_err`,
  `escape/return_cstr_cast_err`. Full suite green (gcc + clang, `-O0` and `-O2`; new paths
  ASan/UBSan-clean).
- **Adjacent pre-existing gap surfaced ‚Äî ‚úÖ RESOLVED 2026-06-14 (see item 3 follow-up).** The
  note originally read: "the assignment-sink escape check catches a stack raw pointer stored
  into a heap struct field (`h->p = &x`) but not a stack slice/str/cstr (`h->name = alloca(c"‚Ä¶")`
  / `(cstr[N]) s` compiles clean)." Re-tested on the current binary: the *non-widened* form is
  already rejected ‚Äî item 3's `assign_dest_prov` walk catches `h->name = (cstr[N]) s` and
  `h->name = alloca(...)` into a non-const field. The real surviving hole was narrower and
  deeper: when the destination is **more `const`** than the value, an implicit `const`-widen
  (`cstr` ‚Üí `const cstr`, slice const-change) ran through `wrap_widen`, which dropped the
  `PROV_STACK` taint ‚Äî defeating *every* sink, not just heap-field. Root-caused and fixed under
  item 3 (`wrap_widen` now propagates provenance); seven negatives + three positives added.

### 7. Nested by-value generic structs corrupt shared type subtrees (pass2.c:3738‚Äì3750, types.c:106) ‚Äî ‚úÖ FIXED 2026-06-15
`struct cell = i: inner<'a>` + a `cell { i = inner { v = 8 } }` literal ‚Üí
`error: unknown type name 'inner_int32'`. Registration shallow-copies (`type_copy` is
`*c = *t`) then `mono_resolve_type_names` mutates the **shared** field stub in place.
Basic pattern; only the pointer-indirection variant is in the test suite.
- **Fix:** deep-copy before canonicalization, or defer all canonicalization to
  `mono_finalize_types`.
- **Resolution (2026-06-15):** deep-copy (the first option). `mangle_type_name`/
  `mono_resolve_type_names` rewrite struct/union/stub **names in place**, and a mono
  instance's `concrete_type` was built with the shallow `type_copy` (`*c = *t`, which aliases
  the `struc.fields` array) or directly from `type_substitute` (which shares *unchanged* field
  subtrees with the template). So canonicalizing a nested generic field name ‚Äî `inner<int32>`
  inside `cell<int32>` ‚Äî mutated a stub that was still the live struct-literal type's field
  (`e->type`, here) or the generic template's field, producing `unknown type name
  'inner__5_int32'` when that corrupted type was later re-resolved. New `type_deep_copy`
  (types.c, declared in types.h) recursively copies every constructor node + a fresh
  fields/variants/params array, sharing only true leaves (primitive singletons, type vars,
  `any*`) and read-only `type_args`; recursion terminates because a field referencing another
  (or the same) generic struct is a `TYPE_STUB`/pointer-to-stub ‚Äî a name + concrete args ‚Äî not
  the embedded definition (by-value self-embedding is rejected as infinite-size). The invariant
  is now **every `concrete_type` is a private deep copy before any in-place canonicalization**:
  the two pass2 sites that canonicalize during type-checking (struct-literal + tuple registration)
  and the four `monomorph.c` discover/build sites deep-copy their `ct` before `mono_resolve`,
  and `mono_finalize_types` deep-copies every `concrete_type` once up front (covering the pass2

### 8. Concrete decls referencing generic instances (monomorph.c:561‚Äì593, codegen.c ~5222‚Äì5290) ‚Äî ‚úÖ FIXED 2026-06-15
- Concrete struct/union fields naming generic instances (`b: box<box<int32> >`) never get

- By-value option-of-generic-struct fields emitted before the option body (incomplete type).

- **Resolution (2026-06-15):** three independent defects, all because CONCRETE top-level decls

  - **8a (raw nested field name).** Root cause was *not* missing discovery over concrete decls

    **dropping a STUB's `type_args`**: it returned the bare base name, so a stub nested inside
    another instance's type args (`box<box<int32>>`, where the inner arg is still an unresolved
    stub) mangled to `box__3_box` instead of `box__12_box__5_int32` ‚Äî the codegen
    `resolve_struct_stub` lookup then missed and fell back to the raw template name `box`.
    Single-level `box<int32>` worked only because its arg is a primitive. Fix: `mangle_type_name`'s

    twice and breaking the interleave name-match. Fix: `collect_types_in_type` canonicalizes a
    stub option-inner to its monomorphized struct before storing, so both forms dedupe and the
    inner carries the mangled name. Covers the never-constructed (`none`-only) case too, since the
    option is collected from the field type itself.

### 9. Non-injective name mangling (types.c:819‚Äì872) ‚Äî ‚úÖ FIXED 2026-06-15
- `TYPE_FUNC` and `any*` both mangle to `"unknown"` ‚Üí distinct instantiations dedupe to one
  C symbol.
- Underscore joining: `pair<foo_bar, baz>` ‚â° `pair<foo, bar_baz>`; user structs named
  `ptr_int32`/`opt_int32` collide with mangled instances. Could *silently merge* if layouts
  coincide.
- **Fix:** structural manglings for fn/any*; injective joining (length-prefixed components
  or escaped separators).
- **Resolution (2026-06-15):** `mangle_type_name` (the per-type leaf mangler) and the two
  joiners (`mangle_generic_name`, `tuple_canonical_name`) in `types.c` are now provably
  **injective** ‚Äî distinct types/instantiations can never produce the same C symbol ‚Äî via two
  devices that exploit the lexer's existing reservation of `__` (double underscore is rejected
  in user identifiers, and module mangling only ever emits `__` *followed by a name segment*,
  which starts with a letter or `_`, never a digit):
  - **Structural tags (fixes the leaf-level collisions).** Every type *constructor* now carries
    a leading `__` tag that no leaf name can forge: pointer `__p`, slice `__l`, option `__o`,
    const `__c`, any* `__y`, fixed-array `__a<N>_`, function `__f<nparams>_‚Ä¶`, plus `char`
    (previously fell through to `"unknown"`) as a leaf and `__err`/`__unk` defensively. So
    `int32*` ‚Üí `__pint32` can never collide with a user struct named `ptr_int32` ‚Üí `ptr_int32`,
    and `TYPE_FUNC` and `any*` (both formerly `"unknown"`) are now distinct and structural.
    Function types mangle their full signature (arity + length-prefixed params + return +
    variadic flag), so different signatures stay distinct. These `__`-tagged strings only ever
    appear *inside* a length-prefixed join component, never as a raw C identifier, so no
    leading-`__` (C-reserved) identifier is emitted.
  - **Length-prefixed joining (fixes the join-level ambiguity).** Both joiners emit
    `base "__" (dec(len) "_" component)*`. The decimal length on each component makes a `_`
    inside one component unconfusable with a component boundary, so `pair<foo_bar,baz>` ‚Üí
    `pair__7_foo_bar3_baz` and `pair<foo,bar_baz>` ‚Üí `pair__3_foo7_bar_baz` are distinct. The
    `base"__"` boundary is itself unambiguous: every arg piece begins with a digit, whereas any
    `__` *within* a module-mangled base is followed by a letter/`_` segment ‚Äî so a base whose
    own name embeds `__<arg-like>` can't be confused with the boundary either.
  - The result stays readable in the common case (`box<int32>` ‚Üí `box__5_int32`). The mangling
    is **`int`-width-agnostic** (pure string construction, no integer-width assumptions) and
    purely a compile-time naming change ‚Äî no generated-C runtime behavior changes, and the
    scheme is an internal implementation detail (no user-facing spec text). Verified injective
    on the three original repros (`box<fn>`/`box<any*>`, `pair<foo_bar,baz>`/`pair<foo,bar_baz>`,
    `box<int32*>`/`box<ptr_int32>` ‚Äî all emitted conflicting C and *failed to compile* pre-fix).
  - Tests (`tests/cases/generics/`): `mangle_fn_anyptr` (fn vs any* coexist; any* round-trips
    through a cast + deref), `mangle_join_underscore` (the two `pair` instances with distinct
    layouts), `mangle_struct_name_vs_structural` (user `ptr_int32`/`opt_int32` vs `int32*`/
    `int32?`), `mangle_fn_arity_nested` (function types of different arity + nested
    `box<box<int32>>`) ‚Äî all four confirmed to fail to compile on the pre-fix binary. Full suite
    green (gcc + clang, `-O0` and `-O2`).

### 14. No instantiation depth limit ‚Üí compiler hangs

### 15. `INT_MIN / -1` and `% -1` SIGFPE (divisor guard checks only 0) ‚Äî ‚úÖ FIXED 2026-06-12 (wrap)
Exit 136, not an FC abort. Spec's "overflow is not applicable" claim for division is wrong
(deliberately left unfixed pending this decision). **Also subsumes the narrow-width division
symptom the audit filed under item 1** (`-128i8 / -1i8` ‚Üí `+128`): that is the same
`min / -1` overflow reaching `int8`/`int16` through C integer promotion. Its visibility is
target-dependent ‚Äî silently wrong on a 32-bit `int` for `int8`/`int16`, but a real SIGFPE/UB
where the FC type is as wide as `int` (`int16` on a 16-bit `int`). So the decision must cover
**all** signed widths `int8`/`int16`/`int32`/`int64`/`isize` uniformly, and a result
cast-back is *not* a valid fix (the trap precedes it).
- **Decision:** define as wrap (`min`, rem `0` ‚Äî what the cast-through-unsigned
  philosophy implies; cheap: extend the existing by-zero guard to also test `== -1` against
  the type's min) vs. abort like /0. Precedent: Rust panics (debug) / wraps (release); C UB;
  Java wraps.
- **Resolution (2026-06-12):** **wrap**, the choice that keeps both standing guarantees true
  ‚Äî the spec's "all signed overflow wraps, eliminating this class of bug entirely" promise,
  and the already-shipped `min * -1 ‚Üí min` (item 1). `a / -1 == a * -1` numerically, so
  aborting one while wrapping the other would be incoherent; matches Java/Go (`/0` aborts,
  `min/-1` wraps). The divisor guard in `codegen.c` now, for **signed** types only, hoists
  the dividend and emits `(_dv == -1) ? <wrapped> : <divide>`: the quotient is a wrapping
  negation through the unsigned counterpart (`(T)(-(UT)nv)`, correct for `min` and every
  width), the remainder is `0`. Unsigned division keeps the plain by-zero-guarded form (an
  all-ones divisor is `max`, not `-1`, and never overflows). The trap precedes any result, so
  this is **`int`-width-agnostic** ‚Äî correct where the FC type is as wide as C's `int` (e.g.
  `int16` on a 16-bit `int`), which a result cast-back could not fix. Divide-by-zero is
  unchanged (still aborts). The compile-time const-fold path (`pass2.c`) already wrapped
  (`ra == -1` guard), so runtime now matches it. Subsumes the `-128i8 / -1i8` symptom filed
  under item 1, uniformly across `int8`/`int16`/`int32`/`int64`/`isize`. Spec updated:
  "Integer overflow" examples + Transpilation note, the `Integer division truncates‚Ä¶` line,
  and the "Division by zero" failure section. Tests: `expressions/div_overflow_wrap` (all
  signed widths, `min/-1`, `min%-1`, non-min negation, unsigned all-ones divisor unaffected;
  UBSan/ASan-clean), `const_eval/div_overflow_min` (fold parity), and the `min/-1` cases
  re-added to `expressions/narrow_arith_wrap` + `narrow_arith_consistency` (deferred there
  under item 1).

### 16. Out-of-range float‚Üíint casts are raw C casts (UB) ‚Äî ‚úÖ FIXED 2026-06-14 (saturate)
`(int32) 1e30` gives three different answers at gcc -O0 / gcc -O2 / clang -O2.
- **Decision:** saturate (helper function, like the `fc_to_size` precedent; matches Rust
  `as` and zig `@intFromFloat` truncating semantics) vs. abort.
- **Resolution (2026-06-14):** **saturate** ‚Äî the choice that makes `(intT) floatexpr`
  *total* (no new abort surface) and keeps FC's "defined behavior over C UB" stance,
  matching Rust `as` and WebAssembly's saturating-truncation conversions. `float_to_int_helper`
  (codegen.c) now routes every float‚Üíinteger `EXPR_CAST` through a per-target-type
  `static inline` helper (`fc_f2i8`‚Ä¶`fc_f2u64`, `fc_f2isize`, `fc_f2usize`) emitted in the
  runtime prelude next to `fc_to_size`/`fc_to_int`, instead of the raw `((int32_t)f)` cast.
  Each helper: `nan ‚Üí 0`; clamp to the target's min/max when the value is at/beyond the range
  edge; otherwise truncate toward zero. All take `double` ‚Äî a `float32` operand promotes
  losslessly, so the bounds and truncation are identical regardless of source width. The
  guard constants are the exact power-of-two boundaries just *outside* each type's range (all
  exactly representable as `double`), so the inner C cast is only ever reached on a value it
  can represent ‚Äî **no residual UB**, verified UBSan/ASan-clean and identical across
  gcc/clang √ó -O0/-O2 (was 134 / 0 / 48 pre-fix). Fixed-width helpers use absolute constants
  (so the routing is **`int`-width-agnostic**); `isize`/`usize` derive their bounds from the
  target's `PTRDIFF_MIN` / `SIZE_MAX`.
  - **Two emission sites (the helper call is not always valid).** The runtime form above is a
    function call ‚Äî illegal in a C *constant expression*. A float‚Üíint cast IS an `is_const_expr`
    form (the `EXPR_CAST` case only excludes str‚Üîcstr), so it reaches **const context**:
    module-**member** `let`s are emitted as C file-scope initializers (`int32_t m__g = ‚Ä¶;`) and
    module-member array literals as `static T _fc_const_backing_N[] = {‚Ä¶}`. (Entry-file-level
    globals are *not* affected ‚Äî their inits hoist into `fc_main` as runtime assignments, the
    helper call is fine there; only module **members** emit at file scope.) Pre-item-16 this
    worked because the raw `((int32_t)3.5)` cast was a C constant; the naive fix `m__g =
    fc_f2i32(3.5);` regresses to *"initializer element is not constant"* on gcc+clang. **Fix:**
    a single `float_to_int_info` table drives both forms, and `float_to_int_emit` picks by
    `g_const_context`: runtime ‚Üí the single-evaluation helper call; const ‚Üí an equivalent
    saturating **ternary** (`(x)!=(x) ? 0 : ((x)<lo ? min : ((x)>=hi ? max : (T)(x)))`), which
    is a valid constant expression with no function call. The ternary evaluates the operand up
    to 4√ó ‚Äî harmless because a const-context operand is pure ‚Äî and emits the operand literally,
    so the float arithmetic is folded by the C compiler at the **target's** semantics (FC does
    not fold floats; const-fold path untouched ‚Äî `const_read_scalar` never reads float
    operands). The dead out-of-range `(int64_t)f` branch does **not** trip gcc/clang
    `-Woverflow` (the controlling compare is a constant). Both forms come from one table, so
    they cannot drift.
  - Spec "Casting" gains a "Float-to-integer casts saturate" subsection (truncate toward zero /
    clamp / `nan ‚Üí 0`, with a float32-promotes note); the implicit-conversions section
    cross-references it. Tests: `casts_widening/float_to_int_saturate` (runtime path ‚Äî all
    widths, both overflow signs, `nan`, `¬±inf`, in-range truncation, `float32` source,
    platform-width `isize`/`usize`, and cast-result-feeds-widening/arithmetic interactions) and
    `casts_widening/float_to_int_const_context` (const path ‚Äî module-member scalars + a static
    backing array, all widths incl. 64-bit/`isize`/`usize`, `nan` via `0.0/0.0`); the const
    test would *fail to compile* on the naive (helper-everywhere) fix and on the original
    raw-cast binary.

### 17. Null-sentinel option conflation (codegen option representation) ‚Äî ‚úÖ FIXED 2026-06-15 (guard `some(p)`)
`some(default(int32*))` silently observes as `none` ‚Äî null is reachable in pure FC via
`default(T*)`, `(T*) 0`, and extern returns. `default((int32)->int32)` invocation is an
unchecked SIGSEGV.
- **Decision:** runtime non-null check in `some(p)` construction for sentinel options
  (abort like unwrap), vs. disallow `default(T*)`/`default(fn)` in pure FC, vs. document
  the conflation.
- **Resolution (2026-06-15):** **guard `some(p)`** (the Swift-school "trap on the bad
  operation" choice). The null-sentinel representation of `T*?`/`any*?`/`cstr?` (bare C
  pointer, `NULL` = `none`) is sound iff a `some` payload is never null. The spec asserted
  that invariant (*"raw `T*` is never null"*) while simultaneously documenting that
  `default(T*)` yields null ‚Äî a contradiction. Rather than ban every null source
  (`default(T*)`, `(T*)0`, zero-init pointer fields, extern returns ‚Äî a large breaking change
  that *still* couldn't be complete, the rejected "partial fix"), the fix guards the one
  injection point: there is **no implicit `T ‚Üí T?` wrapping** in FC, so `some()` is the sole
  place a null pointer can become a `some`. Guarding it catches null from *every* source
  while leaving the representation and C ABI untouched (a C-returned `T*?` never goes through
  `some()`, so null ‚Üí `none` stays the intended encoding). Mirrors item 18's
  static-reject-where-provable + runtime-guard-otherwise + elide-where-provable shape exactly.
  - **Static (pass2 `check_expr` `EXPR_SOME`):** a **provably-null** payload of a
    null-sentinel option is a compile error (`some() of a null pointer is indistinguishable
    from none`) ‚Äî `some(default(T*))`, `some((T*) 0)`, `some(default(any*))`.
  - **Const context (pass2 `const_fold_expr` `EXPR_SOME`):** the guard is a runtime
    statement-expression, not a valid C file-scope constant, so a module-member `some()` of a
    **not-provably-non-null** pointer is rejected with a targeted message (*"some() of a
    possibly-null pointer is not allowed in a constant initializer"*). Provably-non-null
    payloads emit the bare pointer (a valid constant) and are fine. Entry-file globals are
    unaffected (their inits hoist into `fc_main` as runtime assignments ‚Äî same const-context
    boundary as items 16/18).
  - **Runtime (codegen `EXPR_SOME`):** a not-provably-non-null null-sentinel `some(p)` emits
    a statement-expression that evaluates `p` once into a temp and aborts via a new
    cold/`noreturn` `fc_null_some` helper (sibling of `fc_oob`/`fc_neg_len`, registered as a
    backtrace skip-entry) if it is `NULL`, before constructing the option. Plain
    statement-expression, no `alloca` ‚Üí loop-safe (verified at 100 k iters). A generic
    `some(p:'a)` is null-sentinel only once `'a` monomorphizes to a pointer, where the same
    runtime guard fires.
  - **Elision (`ptr_value_provably_nonnull`):** the guard is skipped when the payload can
    never be null ‚Äî `&x`/`&fn`, `p!` (unwrap yields the non-null some-payload; `alloc(...)!`
    is malloc-checked), `c"‚Ä¶"`, and `(T*) <nonzero integer literal>` (e.g. a fixed MMIO
    address). Conservative: anything uncertain (params, field reads, call/extern results) is
    guarded. The predicate (and its `ptr_value_provably_null` sibling) lives in **one place**
    (`codegen.c`, declared in `ast.h` ‚Äî the `interp_is_runtime_sized` pattern) so the
    pass2 reject, the const-context gate, and the codegen elide can never drift.
  - **`int`-width-agnostic:** the guard is a pointer-vs-`NULL` compare and a `noreturn` call;
    no integer-width assumptions.
  - **`default(fn)()` / null function-pointer call ‚Äî left as documented (no code change).**
    A function value is a fat pointer `{fn_ptr, ctx}`; `default(fn)` zeros it and calling it
    dereferences a null `fn_ptr`. This is the *same class* as `*default(T*)` (a null deref),
    which the spec already accepts as programmer responsibility (Default Values: *"Calling a
    zero-initialized function value will crash"*) ‚Äî a loud crash, not silent corruption, and
    not on the option path. A per-indirect-call null check was declined (it taxes every call
    through a function value, against the "static costs over runtime machinery" principle).
  - Spec: the ¬ßConstructing options section gains a "`some` of a null pointer" subsection
    (compile error / runtime guard / elision / const-context restriction); the Transpilation
    note's self-contradiction (*"raw `T*` is never null"*) is corrected to state the sentinel
    is reserved for `none` by the `some()` guard rather than assumed never to occur.
  - Tests (`tests/cases/options/`): four negatives ‚Äî `some_ptr_null_default_err`,
    `some_ptr_null_cast_err`, `some_ptr_null_anyptr_err` (static rejects),
    `some_ptr_null_const_err` (const-context reject); two runtime aborts ‚Äî
    `some_ptr_null_runtime` (laundered null ‚Üí abort 134 at construction),
    `some_ptr_null_generic_runtime` (guard fires through a generic body); four positives ‚Äî
    `some_ptr_nonnull_ok` (all elision forms + a guarded-but-non-null pointer passing),
    `some_ptr_const_ok` (const-context provably-non-null), `some_ptr_loop_ok` (100 k-iter
    loop, no stack growth), `some_ptr_generic_ok` (non-null round-trips through a generic
    pointer option). Full suite green (gcc + clang); new positive paths ASan/UBSan-clean.

### 18. Negative `.len` in raw slice literals defeats bounds checks (pass2.c:4418‚Äì4455) ‚Äî ‚úÖ FIXED 2026-06-15
`int32[] { ptr = ..., len = -1 }` ‚Üí `(uint64_t)idx >= (uint64_t)(-1)` never fires ‚Üí SEGV.
- **Fix:** reject statically-negative `len`; optionally make the runtime check
  negative-len-safe.
- **Resolution (2026-06-15):** both halves, because a static-only fix leaves the
  runtime-computed negative len a live SEGV (partial fix ‚Äî forbidden by the
  completeness principle). The slice-length invariant `len ‚â• 0` is now enforced
  at the **one** construction site (the raw slice literal `EXPR_SLICE_LIT`), so
  every downstream slice keeps the invariant and the hot index/subslice path is
  untouched. (a) **Static:** pass2's `EXPR_SLICE_LIT` rejects a negative integer
  *literal* length (`slice literal length cannot be negative`) and flags a
  provably-non-negative length (`slice_lit.len_nonneg`: an unsigned-typed source,
  or a non-negative literal) so codegen can skip the guard. (b) **Const context:**
  a module-member slice literal emits as a C file-scope initializer with *no*
  runtime guard (a function call isn't a constant expression ‚Äî same constraint as
  item 16), so `const_fold_expr` additionally rejects a *computed* constant that
  folds negative (`0 - 1`, a reference to a negative const), deduped from the
  literal case via `slicelit_len_is_literal`. (c) **Runtime:** codegen wraps a
  non-provably-non-negative len in a statement-expression that evaluates it once
  into an `int64_t` temp and aborts via a new cold/noreturn `fc_neg_len` helper
  (sibling of `fc_oob`/`fc_oob_sub`, registered as a backtrace skip-entry) if it
  is `< 0`, before constructing the slice ‚Äî so a negative runtime len aborts at
  construction instead of silently corrupting every later access. The guard is a
  plain statement-expression (no `alloca`), so it is loop-safe. `int`-width-agnostic
  (the temp is `int64_t`, matching the slice `.len` field; the source widens
  losslessly). Spec ¬ß"Slice construction from raw pointer" gains the non-negative
  requirement (compile error + runtime guard + when the guard is elided). Tests
  (`tests/cases/slices/`): `slice_lit_neg_len_err`, `slice_lit_neg_len_width_err`
  (narrow `-1i8`), `slice_lit_neg_len_const_err` (module-member literal),
  `slice_lit_neg_len_computed_err` (module-member `0 - 1`) ‚Äî all four confirmed to
  compile-and-SEGV on the pre-fix binary; `slice_lit_neg_len_runtime` (runtime
  negative ‚Üí abort 134 at construction); and four positives proving no
  over-rejection (`slice_lit_len_nonneg` literal/unsigned/runtime/zero,
  `slice_lit_len_loop` runtime len in a 1000-iter loop, `slice_lit_len_oob_index`
  a valid runtime-len slice still bounds-checks indexing). Full suite green
  (gcc + clang, `-O0` and `-O2`; new paths ASan/UBSan-clean).

### 19. Evaluation order is unspecified (spec gap) and inconsistent ‚Äî ‚úÖ FIXED 2026-06-15 (left-to-right)
Call arguments evaluate right-to-left under gcc (inherited C unspecified order);
`/` and `%` deterministically evaluate **rhs before lhs** (the by-zero guard temp) ‚Äî
the only deterministic operators, and backwards; `s[f()] = g()` is unsequenced.
- **Decision:** spec left-to-right (modern norm: Rust, Java, C#, Zig) and sequence
  side-effecting operands/args through temps; or explicitly spec "unspecified, as C".
- **Resolution (2026-06-15):** **left-to-right, evaluate-once**, the modern-deterministic
  choice, consistent with FC's already-deterministic interpolation and for-range/iterable
  and with its wart-fixing ethos elsewhere. codegen.c grows a `expr_has_side_effects`
  predicate and a small operand-sequencing helper (`seq_needed`/`seq_hoist`/`seq_restore`):
  where an operand list contains a side-effecting expression, the earlier operands are
  evaluated into temporaries ‚Äî in source order, inside a `({ ‚Ä¶ })` statement-expression ‚Äî
  and the slots rewritten to read them (pure operands stay in place, so the common case is
  untouched; temps are optimized away ‚Üí zero cost). Applied to **binary operands** (left
  before right), **call arguments** (callee first for indirect calls, then args in order),
  **assignment** (target lvalue subexpressions via address-of, then value, then store), and
  **struct/tuple/array-literal fields** (array literals already assigned element-by-element).
  `/` and `%` reordered to hoist the dividend before the divisor (was backwards). `&&`/`||`
  are excluded ‚Äî they already short-circuit left-to-right and their right operand must stay
  conditionally evaluated. A pre-existing multiple-evaluation bug in fixed-array *field*
  assignment (`arr[f()].buf = src` emitted the target object up to four times) was fixed in
  the same pass by hoisting the destination pointer once. The `if `-without-parens shortcut
  for binary conditions (`emit_if_stmt`, value-`if`, match guards) now defers to
  `emit_self_parens`, since a sequenced binary emits `({ ‚Ä¶ })` rather than `(‚Ä¶)`. Spec adds
  an **Evaluation order** subsection under *Operators & Access*. Tests:
  `expressions/eval_order_{call_args,binary,division,assign,aggregates,indirect_call}`.
