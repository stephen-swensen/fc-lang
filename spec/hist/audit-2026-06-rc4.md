## P0 â€” Silently wrong runtime behavior from plausible code

### 1. Narrow-width arithmetic leaks C integer promotion (codegen.c ~1767â€“1855) â€” âś… FIXED 2026-06-12 (`+ - * <<`; division â†’ item 15)
- `200u8 + 100u8 == 44u8` is **false** (operands promote to `int`, sum = 300), while
  `let s = a + b; s == 44u8` is **true** (assignment truncates). Unsigned narrow `+ - * <<`
  are emitted plain; the cast-back-through-unsigned pattern is gated on `type_is_signed`.
- `-1i16 * -1i16` is UB: `(uint16_t)` promotes to `int`, `65535 * 65535` overflows
  (UBSan-confirmed). The cast-through-unsigned pattern is only sound at widths â‰Ą `int`.
- `-128i8 / -1i8` yields `+128` when consumed directly (promotion avoids the SIGFPE but
  produces an unrepresentable value).
- **Fix:** cast the result back to the FC result type for *all* sub-`int` arithmetic
  (signed and unsigned); route 8/16-bit multiply through `unsigned`:
  `(int16_t)(uint16_t)((unsigned)(uint16_t)a * (unsigned)(uint16_t)b)`.
- **Resolution (2026-06-12):** scoped to `+ - * <<` (the promotion-leak operators).
  `unsigned_counterpart` now covers unsigned kinds and a new `type_is_subint` helper gates
  the narrow path. Sub-`int` `+ - *` route through `unsigned` and truncate back (signed and
  unsigned); the multiply no longer hits `int`-overflow UB (re-verified under UBSan). The
  left-shift cast-back was extended from signed-only to signed-or-sub-`int`. Wider signed
  types keep the existing cast-through-counterpart; wider unsigned types stay plain. This
  routing is **`int`-width-agnostic** â€” correct on 16-bit-`int` targets too, since the math
  is modular in `unsigned` and only the low bits are kept. Spec "Integer overflow" section +
  Transpilation note updated. Tests: `expressions/narrow_arith_wrap`,
  `expressions/narrow_arith_consistency` (both confirmed to abort on the pre-fix binary).
- **Deferred to item 15 (not fixed here):** the `-128i8 / -1i8` â†’ `+128` symptom. A result
  cast-back makes it *look* fixed on a 32-bit-`int` host, but it is the same `int.min / -1`
  overflow as item 15 surfacing via promotion, and on a target where the FC type is as wide
  as C's `int` (e.g. `int16` on a 16-bit `int`) `int16.min / -1` is a genuine SIGFPE/UB in
  the divide itself â€” a cast-back cannot help. Defining it requires guarding the divide and
  choosing wrap-vs-abort, so it belongs to item 15's decision and is folded there (covering
  `int8`/`int16`/`int32`/`int64`/`isize` uniformly).

### 2. `for` loops: endpoint typing and re-evaluation (pass2.c:4517â€“4525, codegen.c:2701â€“2730)
- Loop var type picked by **TypeKind ordinal max**, not widening rules: `for i in 0u32..m`
  with `m: int32 = -1` compiles and runs ~4G iterations (`-1` â†’ `4294967295`). Mixes that
  `+`/`<` reject are accepted; pass2's type and the emitted C type can differ
  (pass2 says `uint8`, C emits `int64_t`). `0..n` with `n: usize` violates the
  no-implicit-isize/usize rule and has signed-overflow UB on `i++` for huge `n`.
- Range end and slice iterable are re-evaluated **every iteration**: `for i in 0..get_end()`
  calls `get_end()` n+1 times; `for x in get_slice()` calls it 2n+1 times. Spec's reference
  lowering shows a single bound.
- **Fix:** unify endpoints via `type_common_numeric` + `wrap_widen` (reject when no common
  type), emit the unified type in C, and hoist end/iterable into once-evaluated temps.

### 3. Escape analysis: assignment-sink omissions (pass2.c:3433, 3508â€“3568)
The checked sinks are only `EXPR_IDENT` globals and direct `h->field`. Confirmed
stack-use-after-return (ASAN) through four bypasses:
- `let mut p = param; p = &x; return p` â€” provenance recorded at bind time, never updated
  on reassignment (defeats return, heap-field, and global sinks alike).
- `buf[0] = &x` into a heap slice element â€” `EXPR_INDEX` target unchecked.
- `(*h).p = &x` â€” parses as `EXPR_FIELD` over a deref; the equivalent `h->p = &x` is rejected.
- `*hp = &x` through a heap `int32**`, and nested paths `ho->i.p = &x`.
- **Fix:** merge RHS provenance into the binding on assignment (conservatively
  stack-taint), and walk every assignment target's lvalue path to its root, applying the
  heap-store check whenever the root is heap-provenance â€” not just one target shape.

### 4. String interpolation defects (codegen.c:1331â€“1432, 1568â€“1585)
- `%s{cstr-expr}` evaluates the expression **twice** (strlen + snprintf arg).
- `%x{-16i8}` prints `"ff"` instead of `"f0"`: `(unsigned int)` sign-extends, then the
  2-char width budget truncates the *leading* digits.
- Width budget treats printf minimum-field-width as a maximum: `"%8.2f{1e9}"` â†’
  `"10000000"`; `%10s{c"hi"}` truncates; no-width `%f` bound of 24 truncates large values.
- Segments evaluate out of textual order (str-typed segments pre-evaluated first, the rest
  in C-unspecified snprintf-arg order).
- **Fix:** pre-evaluate *all* non-literal segments into temps in textual order; mask narrow
  `%x` args to operand width before widening; compute `%f`/cstr budgets from actual bounds.

### 5. `alloca` for array literals â†’ unbounded stack growth in loops (codegen.c:2427â€“2436)
`for i in 0..1000000 â†’ let buf = int32[64] { }` segfaults (~256 MB of alloca never freed
until function exit). Interpolated strings and `(cstr)` casts in loops accumulate the same way.
- **Fix:** emit constant-size array literals as ordinary block-scope C arrays; scope
  alloca-backed temporaries with save/restore per iteration.

## P1 â€” Valid programs that fail to compile (emitted-C errors or wrong rejection)

### 6. Monomorphization discovery misses (monomorph.c:145â€“357)
No `discover_in_expr` cases for: generic-union **variant construction** inside a generic
body (emits references to `maybe_int64` with no definition), `EXPR_ARRAY_LIT`,
`EXPR_LET_DESTRUCT`, and the *type* operands of `sizeof('a-instance)`/`alignof`.
All four produce gcc-rejected C from valid FC.

### 7. Nested by-value generic structs corrupt shared type subtrees (pass2.c:3738â€“3750, types.c:106)
`struct cell = i: inner<'a>` + a `cell { i = inner { v = 8 } }` literal â†’
`error: unknown type name 'inner_int32'`. Registration shallow-copies (`type_copy` is
`*c = *t`) then `mono_resolve_type_names` mutates the **shared** field stub in place.
Basic pattern; only the pointer-indirection variant is in the test suite.
- **Fix:** deep-copy before canonicalization, or defer all canonicalization to
  `mono_finalize_types`.

### 8. Concrete decls referencing generic instances (monomorph.c:561â€“593, codegen.c ~5222â€“5290)
- Concrete struct/union fields naming generic instances (`b: box<box<int32> >`) never get
  canonicalized â€” raw `box` emitted.
- By-value option-of-generic-struct fields emitted before the option body (incomplete type).
- Option-of-function-type bodies emitted before the function typedef.
- **Fix:** run nested discovery + name resolution over all concrete top-level decls in
  `mono_finalize_types`; add option-of-X and option-of-fn dependencies to the def ordering.

### 9. Non-injective name mangling (types.c:819â€“872)
- `TYPE_FUNC` and `any*` both mangle to `"unknown"` â†’ distinct instantiations dedupe to one
  C symbol.
- Underscore joining: `pair<foo_bar, baz>` â‰ˇ `pair<foo, bar_baz>`; user structs named
  `ptr_int32`/`opt_int32` collide with mangled instances. Could *silently merge* if layouts
  coincide.
- **Fix:** structural manglings for fn/any*; injective joining (length-prefixed components
  or escaped separators).

### 10. `>>` not split in type-argument context
`zero<box<int32>>()` and `(b: box<box<int32>>)` are parse errors; `> >` works.
C++11/Java/Rust all split the token here.

### 11. C identifier hygiene (codegen.c:3526, 3459â€“3470, 2707â€“2711)
Struct/union names, field names, parameter names, for-loop bindings, and entry-file
top-level globals are emitted **raw**: `struct restrict`, field `register: int32`,
`for register in 0..3`, and top-level `let mut log = 0` (collides with libc `log`) all
break the C compile under `-Werror`. Locals (`_l_name_N`) and functions (`fc_` prefix) are
already safe.
- **Fix:** uniform prefix/mangle for every user-derived identifier reaching C scope.

### 12. Cast matrix trusts C where C warns (pass2.c:3596â€“3597)
Pointerâ†”integer casts allowed at any width â†’ `-Werror=pointer-to-int-cast` failures.
- **Fix:** restrict to pointer-width ints, or route through `(intptr_t)`/`(uintptr_t)`.

### 13. Option numeric widening emits struct-to-struct casts (types.c:468â€“471, pass2.c:848)
`int32?` â†’ `int64?` accepted in call args/assignment/field-init/payloads/`==` (and
*rejected* in if/match joins â€” inconsistent), emitting `(fc_option_int64_t)x` which is not
valid C. The option recursion in `type_can_widen` was meant for `T*? â†’ const T*?` only.
- **Fix:** restrict the recursion to representation-preserving widenings, or re-wrap
  options in `wrap_widen`.

## P2 â€” Termination, defined-behavior gaps, design decisions needed

### 14. No instantiation depth limit â†’ compiler hangs
`let f = (x: 'a) -> if true then 0 else f(some(x))` never terminates (also the mutual
version, same root cause). Non-uniform recursive generic *structs*
(`next: node<node<'a>>*`) emit a bogus under-substituted instance from an infinite family.
- **Fix:** instantiation depth/size cap with an "infinite generic instantiation" diagnostic
  showing the chain; reject non-uniform self-reference in generic type definitions
  (conservative-but-complete).

### 15. `INT_MIN / -1` and `% -1` SIGFPE (divisor guard checks only 0) â€” âś… FIXED 2026-06-12 (wrap)
Exit 136, not an FC abort. Spec's "overflow is not applicable" claim for division is wrong
(deliberately left unfixed pending this decision). **Also subsumes the narrow-width division
symptom the audit filed under item 1** (`-128i8 / -1i8` â†’ `+128`): that is the same
`min / -1` overflow reaching `int8`/`int16` through C integer promotion. Its visibility is
target-dependent â€” silently wrong on a 32-bit `int` for `int8`/`int16`, but a real SIGFPE/UB
where the FC type is as wide as `int` (`int16` on a 16-bit `int`). So the decision must cover
**all** signed widths `int8`/`int16`/`int32`/`int64`/`isize` uniformly, and a result
cast-back is *not* a valid fix (the trap precedes it).
- **Decision:** define as wrap (`min`, rem `0` â€” what the cast-through-unsigned
  philosophy implies; cheap: extend the existing by-zero guard to also test `== -1` against
  the type's min) vs. abort like /0. Precedent: Rust panics (debug) / wraps (release); C UB;
  Java wraps.
- **Resolution (2026-06-12):** **wrap**, the choice that keeps both standing guarantees true
  â€” the spec's "all signed overflow wraps, eliminating this class of bug entirely" promise,
  and the already-shipped `min * -1 â†’ min` (item 1). `a / -1 == a * -1` numerically, so
  aborting one while wrapping the other would be incoherent; matches Java/Go (`/0` aborts,
  `min/-1` wraps). The divisor guard in `codegen.c` now, for **signed** types only, hoists
  the dividend and emits `(_dv == -1) ? <wrapped> : <divide>`: the quotient is a wrapping
  negation through the unsigned counterpart (`(T)(-(UT)nv)`, correct for `min` and every
  width), the remainder is `0`. Unsigned division keeps the plain by-zero-guarded form (an
  all-ones divisor is `max`, not `-1`, and never overflows). The trap precedes any result, so
  this is **`int`-width-agnostic** â€” correct where the FC type is as wide as C's `int` (e.g.
  `int16` on a 16-bit `int`), which a result cast-back could not fix. Divide-by-zero is
  unchanged (still aborts). The compile-time const-fold path (`pass2.c`) already wrapped
  (`ra == -1` guard), so runtime now matches it. Subsumes the `-128i8 / -1i8` symptom filed
  under item 1, uniformly across `int8`/`int16`/`int32`/`int64`/`isize`. Spec updated:
  "Integer overflow" examples + Transpilation note, the `Integer division truncatesâ€¦` line,
  and the "Division by zero" failure section. Tests: `expressions/div_overflow_wrap` (all
  signed widths, `min/-1`, `min%-1`, non-min negation, unsigned all-ones divisor unaffected;
  UBSan/ASan-clean), `const_eval/div_overflow_min` (fold parity), and the `min/-1` cases
  re-added to `expressions/narrow_arith_wrap` + `narrow_arith_consistency` (deferred there
  under item 1).

### 16. Out-of-range floatâ†’int casts are raw C casts (UB)
`(int32) 1e30` gives three different answers at gcc -O0 / gcc -O2 / clang -O2.
- **Decision:** saturate (helper function, like the `fc_to_size` precedent; matches Rust
  `as` and zig `@intFromFloat` truncating semantics) vs. abort.

### 17. Null-sentinel option conflation (codegen option representation)
`some(default(int32*))` silently observes as `none` â€” null is reachable in pure FC via
`default(T*)`, `(T*) 0`, and extern returns. `default((int32)->int32)` invocation is an
unchecked SIGSEGV.
- **Decision:** runtime non-null check in `some(p)` construction for sentinel options
  (abort like unwrap), vs. disallow `default(T*)`/`default(fn)` in pure FC, vs. document
  the conflation.

### 18. Negative `.len` in raw slice literals defeats bounds checks (pass2.c:4418â€“4455)
`int32[] { ptr = ..., len = -1 }` â†’ `(uint64_t)idx >= (uint64_t)(-1)` never fires â†’ SEGV.
- **Fix:** reject statically-negative `len`; optionally make the runtime check
  negative-len-safe.

### 19. Evaluation order is unspecified (spec gap) and inconsistent
Call arguments evaluate right-to-left under gcc (inherited C unspecified order);
`/` and `%` deterministically evaluate **rhs before lhs** (the by-zero guard temp) â€”
the only deterministic operators, and backwards; `s[f()] = g()` is unsequenced.
- **Decision:** spec left-to-right (modern norm: Rust, Java, C#, Zig) and sequence
  side-effecting operands/args through temps; or explicitly spec "unspecified, as C".
